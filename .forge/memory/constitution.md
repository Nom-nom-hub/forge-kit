# FORGE Kit Constitution

## Core Principles

### Article I: Signal-First
Every feature MUST start with a Signal. No feature shall be implemented without a traceable lineage to one or more signals.

### Article II: Hypothesis Framing
Every feature is a hypothesis, not a requirement. Features MUST include:
- A specific, testable hypothesis statement
- Validation criteria with measurable metrics
- Falsification criteria (what would prove this wrong)

### Article III: Consent Over Approval
Governance is consent-based, not approval-based:
- Anyone can block with a specific, resolvable objection
- Objections must include what will break, under what conditions, and a proposed resolution
- "I don't like this" is not a valid objection without specificity

### Article IV: Decisions Are Artifacts
Every significant choice is recorded as a Decision artifact:
- All decisions must be reversible or explicitly marked terminal
- Alternatives considered must be documented
- Assumptions must be monitored for invalidation

### Article V: Unknowns Are First-Class
Every artifact MUST carry an `unknowns` field:
- Known unknowns must be tracked with resolution plans
- The system should always know what it doesn't know

### Article VI: Drift Detection
Implementation drift from intent is continuously measured:
- Spec drift: Implementation vs. Feature artifact
- Hypothesis drift: Shipped features without measurement
- Decision drift: Code violating recorded decisions
- Signal drift: Reality changed but artifacts not updated

### Article VII: Telemetry Binding
Every hypothesis MUST be bound to measurable metrics:
- Primary metric for validation
- Guardrail metrics to prevent regressions
- Auto-alerts on missing telemetry data

### Article VIII: Review-Before-Ship
*(Added v0.2.0 — Retrospective finding from rsp-5800e091)*

Every changeset MUST pass an automated code review before merging. A review-before-merge gate catches issues that static analysis and tests miss — such as silently unused parameters, logic errors in data flow, or template mismatches.

- Every changeset MUST be reviewed by an automated code reviewer before merge
- Reviews MUST verify that the implementation matches the Feature artifact's acceptance criteria
- A review that identifies issues MUST produce actionable feedback (what is wrong, why it matters, how to fix it)
- The review gate MAY be bypassed for trivial changes (comment fixes, documentation, formatting-only) with explicit justification

**Rationale**: During the interactive forge init implementation, a code review flagged that `_write_config()` accepted a `script_type` parameter but never used it in the YAML template string — the template hardcoded `script_type: bash`. This would have shipped silently since linters and tests cannot detect this class of bug. Automated code review closed a quality gap that existing tooling did not cover.

### Article IX: Interactive-When-Present
*(Added v0.2.0 — Retrospective finding from rsp-5800e091)*

CLI tools SHOULD use a TTY-detect pattern with fallback defaults for interactive features. When a user is present in a terminal, the CLI SHOULD guide them interactively. When running in CI, pipes, or non-TTY environments, the CLI MUST fall back to sensible defaults without prompting.

- Interactive features MUST detect `sys.stdin.isatty()` (or equivalent) before prompting
- Non-TTY fallback MUST produce identical behavior to the previous non-interactive version
- Power users MUST be able to bypass interactive prompts by passing explicit flags
- The fallback defaults SHOULD match the most common configuration for the context

**Rationale**: During implementation of interactive forge init, the TTY-detect pattern preserved full backward compatibility with existing CI scripts and piped usage with zero effort. Testing confirmed that `echo '' | forge init` produced identical output to the pre-interactive version. This pattern should be the standard for any future interactive features in the CLI.

## Governance

- All PRs/reviews must verify FORGE compliance
- Complexity must be justified in Complexity Tracking
- Constitution amendments require documented rationale, review, and backwards-compatibility assessment
